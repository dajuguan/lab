# Production 基础设施设计总结（网络相关）

## 1. 我的核心理解（当前）

- 大原则：生产系统很难保证完全无 bug，目标应是“出错可控”，而不是“永不出错”。
- 一线能力：必须优先保证链路具备 `可观测`、`可降级`、`可回滚`，并且足够鲁棒。
- 最小检查清单（接口级）：关键接口至少覆盖以下失败模式：
  - `超时`
  - `不可达`
  - `脏响应`
  - `状态不一致`
  - `慢消费者`
  - `重连抖动`

## 2. E9 视角可能指出的缺口

- 仍偏“故障清单”，缺“接口级合同”：
  - 每个接口需明确输入/输出/错误码/时限/重试策略/幂等语义/降级动作。
- 缺少量化目标：
  - 需要 SLO/SLA、错误预算、端到端时延预算，并拆分到每一跳。
- 缺少故障隔离边界设计：
  - 失败是否跨子系统扩散、blast radius 如何限制、是否有熔断/限流/隔离池。
- 缺少演练与验证闭环：
  - 是否有故障注入、回滚演练、runbook、oncall 判据。
- 缺少恢复判据：
  - 自动恢复条件、人工接管条件、cooldown 与防抖策略未标准化。

## 3. 下一步改进方向（从理念到工程闭环）

1. 为每个关键接口补齐 `RPC/Protocol Contract`（可审计文档化）。
2. 建立 `失败矩阵 -> 动作矩阵`，每类失败都要有检测信号和自动动作。
   示例（简化）：
   - `EL 超时`：信号 = `engine_forkchoiceUpdated` p95 > 2s 且连续 3 次超时；动作 = 切换 `fallback EL`，主 EL 标记 `degraded`，30s 后半开探测恢复。
   - `慢消费者`：信号 = 队列积压持续增长 / lag 错误上升；动作 = 降采样或丢低优先级消息，必要时断开最慢连接，保护主链路。
3. 做 `延迟预算分解`，明确端到端预算在各 hop 的分配。
   示例（精简）：端到端目标 `p95 <= 800ms`，分配为 `Builder 120ms` + `Builder->RB 80ms` + `RB 100ms` + `RB->EL 250ms` + `EL 180ms` + `余量 70ms`。
4. 设计 `降级状态机`（进入/退出条件、阈值、熔断与冷却时间）。
   含义（精简）：把“何时降级、何时恢复”写成确定规则，避免人工拍脑袋。
   示例（精简）：
   - 进入 `Degraded`：`EL 超时率 > 5%` 持续 `60s`。
   - 熔断 `Open`：`连续 3 次` 调用失败后停止打主 EL，仅走 fallback。
   - 半开 `Half-Open`：冷却 `30s` 后放 `5%` 探测流量到主 EL。
   - 退出恢复：探测 `2min` 内成功率 `>= 99%` 且 p95 延迟回到阈值内。
5. 设计 `回滚策略`（流量回滚与数据回滚分离，不可逆操作有保护）。
   含义（精简）：先保服务可用，再处理数据一致性；避免“一键全回滚”扩大事故。
   示例（精简）：
   - `流量回滚`：5 分钟内把 100% 流量切回稳定版本（配置开关 + 灰度比例）。
   - `数据回滚`：仅允许对可逆状态执行（如 checkpoint/replay）；不可逆操作要求双人审批 + 只读保护窗。
6. 固化 `GameDay` 机制，周期性验证观测、降级、回滚是否真实可用。
   含义（精简）：定期“演故障”，验证文档和自动化不是纸面方案。
   示例（精简）：
   - 每两周演练 1 个场景（如 EL 不可达 10 分钟）。
   - 验收标准：告警在 `1min` 内触发；`3min` 内进入降级；`10min` 内完成流量回滚；演练后补齐复盘与 runbook。

## 4. WebSocket 数据面专项（最小规范 + 失败矩阵 + 演练）

### 4.1 数据面协议最小规范（建议模板）

- 消息模型：`payload_id`、`index`、`version`、`base/diff`、`metadata` 必填与类型。
- 顺序规则：同一 `payload_id` 下 `index` 必须连续；`index=0` 必须带 base。
- 去重规则：重复 `(payload_id, index)` 直接丢弃，不重复应用。
- 重连规则：断连后从最新快照恢复；若无法确认连续性，清空 overlay 等待新 `index=0`。
- 背压规则：队列超阈值时优先丢旧/断慢消费者，不阻塞主生产路径。
- 回退规则：overlay miss 或序列异常时，RPC `pending` 回退 `latest`（或 canonical 优先）。
- 观测指标：`lag`、`drop`、`gap`、`reconnect_count`、`queue_depth`、`process_latency`。

### 4.2 失败矩阵（示例 6 行）

- `超时`：信号 = ws read timeout；动作 = 指数退避重连，进入 degraded。
- `不可达`：信号 = connect failed 连续 N 次；动作 = 切 fallback，告警升级。
- `脏响应`：信号 = decode/校验失败；动作 = 丢弃消息并计数，超过阈值触发隔离。
- `状态不一致`：信号 = canonical 对账失败；动作 = 清空 overlay，按 canonical 重建。
- `慢消费者`：信号 = queue depth 持续上升；动作 = 丢低优先级/断开慢连接。
- `重连抖动`：信号 = reconnect 频繁 oscillation；动作 = 熔断冷却窗口 + 半开探测。

### 4.3 GameDay 演练用例（建议最小集）

- 用例 A：上游 WS 断连 10 分钟。
  - 目标：验证重连退避、fallback、生效告警。
- 用例 B：注入乱序消息（跳 index）。
  - 目标：验证序列校验、overlay 清空与自动自愈。
- 用例 C：注入慢消费者。
  - 目标：验证背压策略不会拖垮主链路。
- 用例 D：注入脏消息（解码失败/字段缺失）。
  - 目标：验证丢弃策略、错误计数和隔离阈值。
- 用例 E：canonical/reorg 对账冲突。
  - 目标：验证“canonical 优先 + overlay 重建”闭环。
- 验收标准（统一）：`1min` 内可观测到告警，`3min` 内触发预期自动动作，`15min` 内恢复到可服务状态。

## 5. 失效分类的第一性原理推导（精简）

从第一性原理看，系统“正确”至少要同时满足六个约束：

- `可达`：请求能到达处理点。
- `时限`：结果必须在预算时间内返回。
- `数据完整正确性`：传输与处理过程中的数据不损坏。
- `语义一致正确性`: 状态演进、版本解释、幂等语义保持一致。
- `资源稳定`：系统在有限资源下不失稳(不被队列/CPU/内存拖垮), 可能会导致时限或者语义等出问题。
- `依赖成立`：关键外部依赖处于可用前提内。

对应地，故障分类就是这六个约束的“否定”，因此不是经验罗列，而是约束反推：

### 5.1 可达性失效（违反可达约束）

- 不可达
- 断连
- 部分可达

### 5.2 时序失效（违反时限约束）

- 超时
- 延迟抖动
- 重放/乱序

### 5.3 数据完整性失效（违反数据完整约束）

- 脏数据
- 损坏
- 校验失败

### 5.4 语义一致性失效（违反语义一致约束）

- 状态不一致
- 版本不兼容
- 幂等破坏

### 5.5 容量资源失效（违反资源稳定约束）

- 队列积压
- 慢消费者
- 内存/CPU 耗尽

### 5.6 依赖退化失效（违反依赖前提约束）

- 上游降级
- 局部成功全局失败

补充：六类之间有因果链，不是绝对独立。例如资源失效常先表现为时序恶化，再演化为可达性或语义故障。

## 6. 链路维与阶段维（可靠性视角）

`链路维` 与 `阶段维` 不只是功能拆分，在可靠性上分别回答两个问题：

- 链路维：故障会在哪里跨组件扩散（`不扩散`）。
- 阶段维：故障在生命周期哪个点能被尽早拦住（`早拦截`）。

> 链路维看边界，阶段维看内部

### 6.1 链路维要考虑什么（跨组件）

- 边界合同：超时、重试、幂等、错误码与降级动作是否一致。
- 故障隔离：单链路失败是否会拖垮其他链路（blast radius）。
- 回退路径：fallback 是否真实可用（非纸面），切换是否可观测。
- 依赖治理：上游退化时是否有限流、熔断、冷却和恢复判据。

### 6.2 阶段维要考虑什么（组件内）

- 接收：限流、鉴权、格式检查，防止垃圾流量打满入口。
- 校验：顺序/版本/签名/去重校验，阻止脏数据进入状态机。
- 处理：超时与取消传播，避免卡死与资源泄漏。
- 存储：原子性与一致性，避免部分写入与脏状态残留。
- 返回：稳定错误语义与兼容字段，避免客户端误重试/重放。

### 6.3 一句话原则

- 链路维保“`不扩散`”。
- 阶段维保“`早拦截`”。
