## Bootstrapping
核心思想：既然解密函数本身也是一个电路，那么就可以对“解密电路”再做一次 homomorphic evaluation，从而在**不暴露明文**的前提下，把一个噪声已经很大的密文刷新成一个“新鲜密文”。

它依赖的仍然是同态性质：

```
Enc(a+b)=Enc(a)+Enc(b)
<=>
Eval(add, Enc(a), Enc(b)) = Enc(add(a,b))
进一步 generalize:
Eval(f, Enc(x1), Enc(x2), ...) = Enc(f(x1, x2, ...))
```

对 bootstrapping，更准确的说法不是“换一个 key 重新加密”，而是：
- 同态地执行一次解密电路 `Dec`；
- 再把结果以目标密钥下输出成新的 ciphertext；
- 这样得到的密文表示同一个 message，但噪声被“刷新”了。

概念上可写成：
```
ct = Enc_pk(msg)
z  = Enc_pk'(ct)

把 Dec_sk 看成一个由 +, * 组成的电路，并把 sk 的相关信息也以某种形式提供给 Eval:
Eval(Dec_sk, Enc_pk'(ct)) = Enc_pk'(Dec_sk(ct)) = Enc_pk'(msg)
```

几点说明：
- 输出密钥可以是原来的 key，也可以是新的 key；“换 key”只是某些写法/实现里的效果，不是 bootstrapping 的本质；所以，**同一私钥下也可以反复 bootstrap**
- bootstrapping 的本质是 **refresh noise**，也就是把“快不能解密的密文”变回“可继续计算的密文”。
- 严格来说，`Dec` 不是一个纯公开函数，因为它依赖 secret key；所以实际方案里会提供 bootstrapping key / encrypted secret key 等辅助材料，让服务器能够在不知道 secret key 明文的情况下完成这次同态求值。


## Limitation
常见setup($enc(m)= R*p + r*2 +m; dec(ct) = (ct%p)%2$) 下
说明：
- 下述复杂度和量级主要是按 RLWE/BFV/BGV/CKKS 这类主流 FHE 的工程实现来理解，属于粗略数量级，不同 scheme、参数、是否 batching、是否 bootstrap，差异都可能很大。
- 记 `N` 为多项式阶数（常取 `2^k`），`L` 为模数链/RNS limb 个数，`Q` 为总模数，`log Q` 近似表示“可用噪声预算/可支持深度”的大小。
- 术语直觉：
  - `N`：每个密文多项式有多少个系数，可以粗略理解成“每层向量的长度”。
  - `Q`：当前工作的总模数，很多正确性和噪声预算都和它有关。
  - `Q = q1 * q2 * ... * qL`：工程实现里通常不会直接在一个超大 `Q` 上算，而是拆成 `L` 个较小模数分别算，这就是 RNS 表示。
  - `RNS limb`：上面每个小模数 `qi` 对应的一个分量，可理解成一“段”模数。
  - `模数链`：`Q_L -> Q_{L-1} -> ... -> Q_1` 这一串逐层下降的模数；每做若干次乘法/`rescale`/`mod switch`，通常会消耗掉一层。
  - 所以 `LN` 就是 `L * N`，表示“`L` 个分量、每个分量有 `N` 个系数”这一类总工作量。

- 计算成本很高
  - 为什么：明文计算只需要对小整数/浮点数做普通运算；FHE 里做的是“带噪声的密文多项式/向量运算”，底层通常要做大模数上的加减乘、NTT/FFT、key switching、relinearization 等操作。
  - 复杂度：
    - ciphertext add: 逐系数相加，粗略是 `O(LN)`。
    - ciphertext mul: 多项式乘法若用 NTT，粗略是 `O(LN log N)`；再加上 relinearization / key switching，实际常数非常大。
    - rotation / key switching: 通常也是这一类最贵操作之一，常按 `O(LN log N)` 量级看待。
  - 量级：
    - 同样一个 `+`，密文加法常见比明文加法慢 `10^2 ~ 10^4` 倍。
    - 同样一个 `*`，密文乘法常见比明文乘法慢 `10^3 ~ 10^5` 倍。
    - 在很多实现里，真正贵的不只是乘法本身，而是乘法后紧跟的 relinearization / rescale / key switching。
  - 结果：同样一个 `+` 或 `*`，在密文上会被展开成很多次底层大整数运算，所以延迟和吞吐都远差于明文计算。尤其乘法远比加法贵，因为乘法后通常还伴随额外的维度膨胀和降维处理。

- 密文大小很大
  - 为什么：密文不再是“一个数”，而是为了隐藏明文并保留同态性质，需要携带多个环元素/向量分量以及足够大的参数空间来容纳噪声。
  - 复杂度：一个 fresh ciphertext 往往包含 `2` 个环元素，大小粗略可写成 `O(N log Q)` bits；乘法后若未 relinearize，临时会膨胀到 `3` 个甚至更多分量。
  - 量级：
    - 一个明文 `u32/u64` 只有 `4~8B`，对应 FHE 密文常见会膨胀到 `10KB ~ MB` 量级。
    - evaluation key / relinearization key / galois key 往往比单个 ciphertext 更大，常见是 `MB ~ GB` 量级的额外内存占用来源。
  - 结果：一个很小的明文 bit、整数，甚至一个布尔值，都会膨胀成远大于原文的数据结构；如果还要附带 evaluation key / bootstrapping key，整体存储和传输成本会继续上升。
  - 补充：如果使用 SIMD batching，一个 ciphertext 可以打包很多 slot，单个数据的均摊成本会下降；但“单个密文本身很大”这个事实并不会变。

- 可计算深度有限，本质上受 noise/error 增长约束
  - 为什么：这类方案解密依赖“明文项”和“噪声项”还能被区分开。每做一次同态运算，尤其是乘法，噪声都会继续累积；乘法会把两个带噪声的项相乘，所以增长速度比加法快得多。
  - 复杂度/关系：
    - 若要支持乘法深度 `D`，通常需要准备长度约为 `D` 的 modulus chain，所以常写成“支持深度和 `L` 成正相关”，即 `D = O(L)`。
    - 而 `L` 增大意味着 `Q` 增大；为保持同样安全级别，通常又要把 `N` 继续加大。所以支持更深电路，不是线性加一点参数那么简单，而是会连带放大时间和空间成本。
  - 量级：
    - 不做 bootstrapping 的 leveled FHE，工程上常见只支持大约 `5 ~ 30` 层乘法深度；再深就需要明显更大的参数。
    - CKKS 一类近似计算常也只能承受十几到几十层 rescale/mul，具体取决于精度预算。
  - 结果：电路深度不能无限增加。只做少量乘法的场景可以使用 leveled FHE；一旦乘法层数太深，噪声接近甚至超过可解密阈值，结果就会解不出来或解密错误。
  - 对应补救：可以用 bootstrapping 把“快失效的密文”刷新成“新密文”，相当于重置噪声预算；但 bootstrapping 本身又非常贵，往往是系统瓶颈。
  - bootstrapping 量级：
    - 复杂度通常至少不低于一次重型 key switching + 多轮多项式近似/变换，可粗略看成远高于普通 mul 的操作。
    - 工程上常见比一次密文乘法再慢 `10^2 ~ 10^4` 倍；若和明文基本运算相比，慢 `10^6` 甚至更高都不奇怪。

- 参数选择很难，而且安全性/性能/正确性互相制约
  - 为什么：参数太小，噪声容易溢出或者安全性不够；参数太大，计算和密文尺寸又迅速变重。真实系统里需要同时平衡安全级别、支持的乘法深度、明文空间大小、吞吐和延迟。
  - 结果：FHE 不是“算法写完就行”，而是很依赖参数工程。很多方案在理论上可行，但参数一旦按安全要求放大，实际性能就会明显恶化。

- 精度问题
  - 为什么：很多 FHE 方案天然更适合整数/bit 运算。对实数计算时，通常要把浮点数编码成定点数或近似多项式形式；编码、缩放、舍入、rescale 过程中都会引入误差。
  - 结果：不擅长需要高精度连续数值的场景。像除法、比较、非线性函数（`exp/log/sin`）通常都要改写成多项式近似，而近似误差会和同态计算误差叠加。

- 条件分支和随机访问不自然
  - 为什么：密文内容不可见，服务器不能直接根据“密文里的真实值”走 `if/else` 或按下标随机访问内存，否则就泄露信息。
  - 结果：很多看似简单的控制流，需要改写成数据流电路，例如把分支写成 `select/mux`，把查表写成全表扫描或特殊协议。这会进一步放大计算量。

总结：
- FHE 的核心代价来自“为了在不解密的前提下继续计算，必须把明文运算提升为高维、带噪声、可控的代数结构运算”。
- 因此它最主要的局限可以归纳为 4 个词：慢、大、浅、难。
  - 慢：运算慢，尤其乘法和 bootstrapping。
  - 大：密文和辅助 key 很大。
  - 浅：可直接支持的乘法深度有限。
  - 难：参数、精度、程序改写都比较复杂。
- 如果一定要用一句更工程化的话概括：
  - 单次运算成本大致从明文的 `O(1)` / 小整数运算，变成密文上的 `O(LN)` 或 `O(LN log N)`。
  - 单个数据大小大致从 `B` 级别，膨胀到 `KB ~ MB`。
  - 可直接支持的乘法深度通常只有几十层以内；想继续做，就要付出更大的参数和 bootstrapping 成本。
